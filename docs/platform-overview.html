Monorepo Folder Structure
Click any folder to see why it exists, what it contains, and how it connects to other layers.
Why a monorepo? All code — frontend, backend, shared contracts, infrastructure, and documentation — lives in a single repository. This means a single Pull Request can change the API response format and the frontend that consumes it in one atomic commit. No cross-repo coordination, no version mismatches.
apps/ — Deployable Services
Two services that run in production: the Next.js frontend users interact with and the FastAPI backend that processes data. Each uses its own language toolchain (Turborepo for TS, uv for Python) but they deploy together.
contracts/ — The Bridge
Language-neutral JSON files that both apps consume. This is the only folder truly shared between Python and TypeScript. Tenant configs, auth provider settings, and JSON Schemas live here. When you change a feature flag, both apps see it.
packages/ — TS-Only Libraries
Shared React components and auto-generated TypeScript types. These are only used by the frontend — the Python API cannot import them. The API contract flows the other direction: FastAPI generates OpenAPI → TS types are generated from it.
data/ — Database Definitions
Snowflake schemas and Mongo cache configs. These define the data structures the API layer queries against. Changes here ripple up: schema change → API adapts → frontend types regenerate.
infra/ — How It Runs
Dockerfiles build the container images. CI pipelines test both languages and validate all tenant configs. Terraform provisions the AWS infrastructure. This layer wraps everything above into deployable, testable artifacts.
docs/ — Why We Did It
Architecture Decision Records capture the reasoning behind choices (not just what was chosen). Runbooks document operational procedures. User-facing docs ship inside the app container so each client sees docs matching their version.

Multi-Tenant Configuration
Two config sources merge at startup. Explore the config split details or trace the full configuration flow from source to runtime.
Config Split
Configuration Flow
The Config Split — why two sources? Financial clients require that credentials never appear in source control. But feature flags and branding need PR review and staging validation before they reach production. So config is split by concern: non-secret config lives in the repo (reviewable, testable), secrets live in AWS Secrets Manager (secure, auditable). At container startup, the app merges both into a single runtime config object.
Repo JSON — Developer Managed
What: Feature flags, branding (logo, colors, theme), UI layout, nav items, auth provider choice, Snowflake account ID (not password).
Where: contracts/tenant-configs/millennium.json
Changed by: Developer edits file on feature branch → PR review → CI validates against schema → merge to staging → tested → promoted to main.
Why here: These are not sensitive. They need review, version history, and testing before reaching clients.
AWS Secrets Manager — Ops Managed
What: Snowflake password, database connection strings, API keys, auth provider secrets, cache connection strings.
Where: AWS Secrets Manager, one secret per client per environment (dev/staging/prod).
Changed by: Tech lead or DevOps via AWS console/CLI. IAM role per container restricts access to only that client's secret.
Why here: These are sensitive credentials. Never in source control. AWS provides rotation, audit trail, and encryption at rest.
Runtime Merge — At Container Startup
Step 1: Container starts, reads contracts/tenant-configs/{client}.json from disk (baked into image).
Step 2: Fetches secret from AWS Secrets Manager via IAM role.
Step 3: Merges both into a single config object. App code reads from this merged object — never from either source directly.
Result: config.features.bulk_export comes from repo JSON. config.snowflake.password comes from Secrets Manager. App doesn't know or care which source.

▸ Terminology Glossary — for executive and non-technical readers
Monorepo
A single code repository that contains all of the project's code (frontend, backend, infrastructure) instead of splitting them into separate repositories. Simplifies coordination and ensures everything stays in sync.
Container / Docker
A lightweight, portable package that bundles an application with everything it needs to run. Think of it as a self-contained box that runs the same way everywhere — on a developer's laptop or in production.
Container-per-client
Each client gets their own running copy of the application (container), even though they all use the same underlying code. This provides data isolation — one client's container cannot access another client's data.
IAM Role
An AWS security identity that controls what resources a container can access. Each client's container gets a unique IAM role that only permits access to that client's secrets and data.
AWS Secrets Manager
A secure vault service from Amazon that stores passwords, API keys, and connection credentials. The application retrieves these at startup instead of storing them in code files — critical for financial compliance.
Feature Flags
On/off switches in configuration that control which features are available to which clients. Allows the same codebase to show different capabilities per client without separate code.
Tenant
A client or customer who uses the platform. In a multi-tenant system, multiple tenants share the same software but each sees only their own data and configuration.
FastAPI
A modern Python framework for building web APIs (the backend). It automatically generates documentation and is designed for high performance.
Next.js
A React-based framework for building the web frontend (what users see and interact with in their browser).
ShadCN
A pre-built component library (buttons, tables, forms, etc.) for React applications. Provides a consistent, professional look without designing every element from scratch.
Snowflake
A cloud-based data warehouse where all client financial data is stored. Each client gets a separate Snowflake account for complete data isolation.
OpenAPI / API Contract
A standardized description of what the backend API offers — like a menu. The frontend reads this to know exactly what data it can request and what format it will arrive in.
Turborepo
A build tool that speeds up development by intelligently caching and only rebuilding what changed. Manages the TypeScript/JavaScript side of the monorepo.
uv
A fast Python package manager that handles installing and managing Python dependencies. Manages the Python/FastAPI side of the monorepo.
PR (Pull Request)
A formal request to merge code changes into the main project. Other team members review the changes before they're approved — like a quality gate before changes go live.
CI/CD Pipeline
Automated processes that build, test, and deploy code whenever changes are made. Catches errors before they reach production and automates the release process.
ADR (Architecture Decision Record)
A written document that captures an important technical decision, the options considered, and why the chosen approach was selected. Creates an audit trail for architectural choices.
Linear
A lightweight project management tool (replacing Jira) used to track tasks, bugs, and feature work across all teams.
Component Registry
A central catalog that maps component names (like "DataGrid" or "Chart") to actual UI code. When configuration says "show a DataGrid here," the registry looks up and renders the right component.
Schema Validation
Automated checking that configuration files follow the expected structure. Prevents malformed config from breaking the application — like spell-check but for data structure.
Load Balancer
A traffic director that routes incoming requests to the correct client container based on who's making the request. Ensures each client reaches their own dedicated environment.
Lambda
AWS Lambda — small, individual functions that run on-demand. The previous KKR architecture used many separate Lambda functions per client, creating duplication. Being replaced by containers.
Apache Airflow
An open-source workflow orchestration platform that schedules and monitors data pipelines. Pipelines are defined as DAGs (directed acyclic graphs) in Python, with built-in retry logic, dependency management, and a web UI for monitoring.
DAG (Directed Acyclic Graph)
A pipeline definition in Airflow where tasks have dependencies that flow in one direction (no circular dependencies). For example: ingest data → process dimensions → process GL → warm cache. Each step waits for the previous one to complete.
CeleryExecutor
Airflow's production execution mode that distributes pipeline tasks across multiple worker processes using Redis as a message broker. Enables parallel execution of tasks across different tenants.
